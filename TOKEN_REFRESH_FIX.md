# Token 无限循环刷新问题修复

## 问题描述

前端应用在刷新页面时出现无限循环的 token 刷新请求，导致：
- 后端日志显示大量 token 刷新请求
- JWT 认证失败错误
- 用户体验差，页面无法正常加载

## 根本原因分析

应用中存在多个地方同时检查和刷新 token，导致并发刷新请求：

1. **HTTP 请求拦截器** (`src/utils/http.ts`)
   - 每个请求前都调用 `tokenManager.autoRefreshIfNeeded()`
   - 造成大量并发刷新请求

2. **路由守卫** (`src/router/index.ts`)
   - 访问需要认证的路由时检查 token 过期并刷新
   - 与其他刷新逻辑冲突

3. **应用初始化** (`src/utils/appInit.ts`)
   - 应用启动时检查 token 过期并刷新
   - 页面可见性变化时也会刷新 token

4. **Auth Store** (`src/store/auth.ts`)
   - `checkAuth()` 方法中也会刷新过期的 token

5. **自动刷新定时器** (`src/utils/tokenManager.ts`)
   - 每分钟检查一次，过于频繁

## 修复方案

### 1. 简化 HTTP 拦截器逻辑
- **移除**请求拦截器中的自动刷新逻辑
- 只添加 token 到请求头，不主动刷新
- **改进**响应拦截器中的 401 错误处理，增加更详细的日志

### 2. 优化路由守卫策略
- **简化**认证检查逻辑
- token 过期时直接跳转登录页，不尝试刷新
- 避免在路由层面进行 token 刷新

### 3. 调整应用初始化行为
- 初始化时发现 token 过期直接清除，不刷新
- 页面可见性变化时只检查不刷新
- **修复** `checkAuth()` 调用，改为使用本地状态检查

### 4. 修改 Auth Store 检查逻辑
- `checkAuth()` 方法中不再刷新过期 token
- token 过期时直接清除认证状态

### 5. 修复 SecurityMiddleware 组件
- **移除**组件中的 `checkAuth()` API 调用
- **简化**定时检查逻辑，只检查本地 token 状态
- 将检查频率从 30 秒改为 5 分钟

### 6. 修复 API 端点不一致问题
- **统一** `/api/user/me` 端点的使用
- 修复 `api.ts` 中硬编码的 `/api/auth/me` 路径

### 7. 优化 TokenManager 刷新逻辑
- **修复**刷新请求中的认证头，使用 `accessToken` 而不是 `refreshToken`
- **增加**刷新频率限制（30秒内最多一次）
- **增加**重试次数限制（最多3次）
- **改进**并发控制和错误处理

### 8. 改进 autoRefreshIfNeeded 方法
- token 已过期时不尝试刷新，返回 false
- 只在即将过期且无进行中刷新时才刷新
- 增加并发控制

## 修复后的 Token 刷新策略

### 主要刷新触发点：
1. **HTTP 响应拦截器**：收到 401 错误时尝试刷新并重试请求
2. **自动刷新定时器**：每 5 分钟检查一次，在即将过期时刷新
3. **手动触发**：用户主动操作时通过 API 调用刷新

### 防重复机制：
- 使用 `refreshPromise` 确保同时只有一个刷新请求
- 多个并发请求会等待同一个刷新 Promise

### 失败处理：
- 刷新失败时清除所有 token
- 自动跳转到登录页面
- 停止后续的自动刷新尝试

## 测试建议

1. **页面刷新测试**：刷新页面后检查是否还有无限循环
2. **Token 过期测试**：等待 token 过期后检查自动刷新是否正常
3. **并发请求测试**：同时发起多个 API 请求，确保只有一次刷新
4. **网络异常测试**：模拟网络错误，确保错误处理正确

- ✅ 减少不必要的网络请求
- ✅ 提升用户体验和页面加载速度
- ✅ 降低服务器负载
