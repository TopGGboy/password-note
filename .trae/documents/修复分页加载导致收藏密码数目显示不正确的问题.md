# 修复分页加载导致收藏密码数目显示不正确的问题

## 一、问题分析

### 根本原因
目前 `favoriteCount` 是通过前端过滤本地 `entries` 数组计算出来的：
```typescript
const favoriteCount = computed(() =>
  entries.value.filter(entry => entry.favorite).length
)
```

由于使用了分页加载，`entries.value` 数组只包含当前已加载的条目，而不是所有条目：
- 初始加载时，只加载了第一页的数据，所以 `favoriteCount` 只计算了第一页中的收藏条目
- 当用户向下滚动时，会加载更多数据，`entries.value` 数组变大，`favoriteCount` 也会随之增加

这导致收藏密码数目显示不正确，因为它只反映了已加载条目中的收藏数量，而不是所有条目中的收藏数量。

## 二、解决方案

### 方案1：API返回总收藏数量
**优点**：最准确，性能最好
**缺点**：需要后端支持

1. 修改API响应，在分页信息中包含 `totalFavorites` 字段
2. 在前端添加 `totalFavorites` 状态，用于存储总收藏数量
3. 在 `fetchEntries` 和 `searchEntries` 函数中，从API响应中提取 `totalFavorites` 并赋值给 `totalFavorites.value`
4. 修改 `favoriteCount` 计算属性，直接使用 `totalFavorites.value` 而不是前端过滤计算

### 方案2：前端全量加载数据
**优点**：不需要后端支持，实现简单
**缺点**：性能较差，特别是当数据量较大时

1. 修改 `fetchEntries` 函数，一次性加载所有数据
2. 移除分页加载逻辑，改为直接获取所有条目
3. 修改 `favoriteCount` 计算属性，基于完整数据计算收藏数量

### 方案3：初始加载时获取所有收藏条目
**优点**：性能较好，不需要后端支持
**缺点**：实现较复杂

1. 初始加载时，调用专门的API获取所有收藏条目
2. 存储所有收藏条目的ID
3. 根据收藏条目ID计算总收藏数量
4. 当用户切换收藏状态时，更新收藏条目ID集合

## 三、推荐方案

考虑到项目的实际情况，我推荐使用**方案1**，因为它最准确、性能最好。如果后端暂时不支持，可以先使用**方案2**作为临时解决方案，后续再迁移到**方案1**。

## 四、实现步骤

### 方案1实现步骤

1. **修改 `usePasswordEntries.ts`**：
   - 添加 `totalFavorites` 响应式状态
   - 在 `fetchEntries` 和 `searchEntries` 函数中，从API响应中提取 `totalFavorites`
   - 修改返回值，包含 `totalFavorites`

2. **修改 `Passwords.vue`**：
   - 从 `usePasswordEntries` 中获取 `totalFavorites`
   - 修改 `favoriteCount` 计算属性，直接使用 `totalFavorites`

### 方案2实现步骤

1. **修改 `usePasswordEntries.ts`**：
   - 修改 `fetchEntries` 函数，一次性加载所有数据
   - 移除分页加载逻辑
   - 修改 `hasMore` 计算属性，始终返回 `false`

2. **修改 `PasswordEntriesList.vue`**：
   - 移除加载更多相关逻辑

## 五、预期效果

修复后，收藏密码数目将显示所有条目中的收藏数量，而不是仅已加载条目中的收藏数量。用户在初始加载时就能看到准确的收藏密码数目，不会随着滚动加载而变化。
